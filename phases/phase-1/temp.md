# Phase I – Core Todo Backend

## Objective
Implement a basic Todo backend using Spec-Driven Development.

## Todo Entity
- id: UUID
- title: string (required)
- description: string (optional)
- status: pending | completed
- created_at: timestamp
- updated_at: timestamp

## API Endpoints

### Create Todo
POST /todos

### Get Todos
GET /todos

### Update Todo
PUT /todos/{id}

### Delete Todo
DELETE /todos/{id}

## Persistence
- SQLite database
- Data must persist across restarts

## Error Handling
- Invalid input → 400
- Missing Todo → 404

## Acceptance Criteria
- Todos can be created, updated, deleted
- Data persists
- API returns JSON


# Evolution of Todo – Engineering Constitution

## Core Principles
1. Spec-Driven Development is mandatory.
2. Humans must not write implementation code.
3. All code must be generated by an LLM (Gemini).
4. Specs define all behavior.
5. No undocumented behavior is allowed.

## Development Rules
- Follow specs strictly.
- No extra features.
- Code must be production-ready.

## AI Rules
- Gemini generates all code.
- Prompts must include full spec and acceptance criteria.

## Technology Constraints
- Backend language chosen by Gemini.
- AI integrations defined in later phases.
- Docker & Kubernetes required in later phases.

## Documentation Rules
- Each phase must have a spec.
- Acceptance criteria required.
